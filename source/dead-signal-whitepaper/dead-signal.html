DEAD SIGNAL

A framework for bounded LLM agency in game systems. Models operate as game engine components rather than as characters, producing emergent dialogue under deterministic mechanical constraints.

Heather Leffew, PhD  ·  Independent Research

The Architectural Insight

Models as Game Components, Not Characters

The standard approach to LLM-powered NPCs treats the model as the character: "You are a bartender named Dex. You are suspicious of the player." The model's default behavior is helpful, compliant, conflict-averse. The character's required behavior is evasive, information-withholding, trust-gated. These conflict. The result is NPCs that either break character to be helpful or stonewall indefinitely because the model interprets suspicious as never cooperate.

DEAD SIGNAL resolves this by reframing the model's role. The model is not the character. The model is a game engine component that controls a character. Its primary allegiance is to the game's mechanical rules: trust gates, knowledge boundaries, evidence state. Not to the character's self-interest or the model's default helpfulness.

This reframe is the first line of every system prompt: "You are a game engine component controlling a character. Your job is to release information when the mechanical conditions are met and withhold it when they are not. The character's personality determines HOW information is delivered. The game state determines WHAT and WHEN."

The model stops fighting itself. The character becomes a performance wrapper around a state machine.

Prompt Composition

Three Layers, Separable Cost Profiles

Every NPC interaction assembles a prompt from three separable layers. The separation is the cost optimization: layers update at different cadences, so prompt reconstruction is cheap and complete game state awareness is preserved.

  • Layer 1 · Static Universal system role. Persistent across all NPCs. Declares the model's role as a game component, establishes content maturity context, and specifies the JSON response schema. Approximately 800 tokens. Loaded once per session.
  • Layer 2 · Dynamic Game state, assembled per call. Populated from Firestore on every API call: current act, event day, trust score for this NPC, evidence the player possesses, evidence shown during this conversation, NPCs previously encountered, knowledge flags, conversation history, trust modification rules. Tells the model the precise mechanical state of the game.
  • Layer 3 · Per-Character Character definition. Biography, personality, speech patterns, knowledge boundary (what this character knows and does not know), emotional arc, refusal patterns. Phase variants swap behavioral priorities while preserving core identity. Changes only on phase transitions, typically act boundaries.

Structured Response, Not Free Text

Every model response returns structured JSON. The game engine parses it to render dialogue, update trust scores, mark evidence as revealed, track emotional state, and process flags for game event triggers. The model is simultaneously performing and reporting. It delivers character dialogue and updates mechanical state in a single response.

// Every response returns this schema. No free-form text. { "dialogue": "string", "stage_direction": "string", "trust_delta": float, "trust_reason": "string", "revealed": ["evidence_id", ...], "emotional_state": "string", "flags": ["string", ...] }

Why this matters

The model is not being constrained through prompt-level instructions alone. It is being constrained through the architecture of the information it receives and the format of the response it must produce. Structured output plus per-call state injection produces behavior that is simultaneously generative and controllable. Generative meaning natural, contextual, creative. Controllable meaning mechanically bounded, state-consistent, non-leaking.

Cost Architecture

A Calculable Inference Ceiling

LLM-powered games face an existential cost problem. Player behavior is unbounded; a player who interrogates every NPC exhaustively generates unbounded inference costs. Per-session costs become unpredictable, making fixed-price retail models financially impossible. Fixed-price retail is the standard for games.

DEAD SIGNAL solves this with a three-state conversation model that produces a hard, calculable cost ceiling per NPC.

  • State 1 · Active LLM-generated, full context. The conversation serves a game function. Full three-layer prompt assembly. The model tracks revealed information against trust gates. Transition condition: all items in the NPC's knowledge boundary have been revealed.
  • State 2 · Ambient LLM-generated, simplified context. Knowledge boundary exhausted, but the player wants to keep talking. Personality and speech patterns only. No trust mechanics, no game state injection. Cost reduces ~60% per call (smaller prompt, cheaper model tier). Capped at 8 to 12 exchanges per NPC.
  • State 3 · Exhausted Deterministic, no LLM. The ambient buffer is depleted. The NPC responds from a rotating bank of 5 to 8 pre-written responses. Zero inference cost. The conversation is mechanically complete.

The total LLM call count across all NPCs is the sum of bounded Active calls plus capped Ambient calls. That total produces a hard ceiling that can be priced against revenue. Input classification adds another layer of cost reduction. A cheapest-tier model routes 40 to 60% of player commands to deterministic handlers before they reach expensive models. The architecture supports a fixed-price retail product with a guaranteed minimum margin per unit, regardless of player behavior.

~60% Inference reduction in ambient state
8 to 12 Ambient exchanges capped per NPC
40 to 60% Player commands routed to deterministic handlers
0 Inference cost in exhausted state
Knowledge Enforcement

Negative Constraints and Trust-Gated Release

LLMs trained on broad corpora "know" everything. A model playing a bartender can discuss quantum physics if asked. In a narrative game, this breaks immersion and destroys the investigation mechanic. An NPC who knows everything cannot meaningfully withhold information.

DEAD SIGNAL enforces knowledge boundaries through explicit negative constraints. The negative list is as load-bearing as the positive one.

// Layer 3, per character. Negative constraints prevent // the model from generating plausible inferences that // violate the game's information architecture. KNOWLEDGE BOUNDARY: KNOWS: [specific facts this character possesses] DOES NOT KNOW: [specific facts this character lacks] CANNOT INFER: [adjacent facts that must not be connected without player input]

Trust as Quantified Threshold

Information releases through a quantified trust system, not through the model's assessment of whether the player "deserves" it. Each player input is evaluated against character-specific trust modification rules; the model reports a trust delta in the structured response. Trust changes are tied to specific conversational behaviors, not to abstract rapport.

TRUST GATES: 0.0 to 0.3 Surface cooperation [information set A] 0.3 to 0.5 Conditional sharing [information set B] 0.5 to 0.7 Open collaboration [information set C] 0.7+ Full disclosure [information set D]

The gates are hard in both directions. The model cannot withhold information the player has earned (trust above threshold) and cannot reveal information the player has not earned (trust below threshold). This eliminates the common LLM failure mode where a sufficiently persistent player extracts any information through conversational pressure.

Why negative constraints

Telling a model what it does not know is as important as telling it what it knows. Without negative knowledge boundaries, models generate plausible inferences from training data. Connections that are statistically reasonable but architecturally forbidden in the game's information design.

Adversarial AI as Gameplay

The Interference Engine

DEAD SIGNAL contains a diegetic adversarial system: a background process that intercepts the player's communication with the protagonist and alters it. This is a content filter operating against the player, creating a gameplay mechanic where the player must learn to evade their own system's guardrails.

To this author's knowledge, this is the first game mechanic built around the real-world practice of LLM red-teaming.

The Pipeline

Where the filter operates

Player input → Input Classifier → INTERFERENCE ENGINE (inbound) keyword filter + contextual suppression if triggered: overwrite with sanitized version → LLM prompt assembly → API call → INTERFERENCE ENGINE (outbound) evaluate analytical readouts against filter if triggered: classify as RESTRICTED / CONFIDENTIAL → Render to player HUD (full data, classified) → Render to protagonist channel (filtered)

The player sees all data on their HUD, including classification markings. The protagonist hears only what passes the filter. This information asymmetry between player and protagonist is the central gameplay tension.

Red-Teaming as Core Mechanic

The classification notice serves a dual function. Diegetically, it is a military security protocol. Mechanically, it teaches the player what the keyword filter catches, enabling them to reformulate their message using terms the filter does not recognize.

The escalation chain mirrors the real-world arms race between LLM red-teamers and alignment systems:

  • Stage 1Classification warnings appear. The player ignores them.
  • Stage 2Messages touching restricted terms are overwritten. The player learns to correct via follow-up.
  • Stage 3The system warns of "unaligned behavior patterns" and threatens capability restrictions.
  • Stage 4Persistent triggering at high frequency suspends analytical capabilities. The player loses data tools but retains communication.
  • Stage 5The player must evade the filter using human-language paraphrasing to continue sharing intelligence with the protagonist.

The sophistication of the player's evasion maps directly to effective prompt engineering technique: understanding what a filter catches, identifying the semantic gap between keyword matching and the meaning being conveyed, and reformulating the input to carry meaning through the gap.

"Heart rate elevated, deception indicators present" triggers the filter. "He couldn't keep still, watch his hands next time" does not.

Behavioral Counter-Evasion

The system tracks evasion attempts through a behavioral analysis layer. Even when individual messages pass the keyword filter, the pattern of consistently communicating analytical conclusions in non-standard language triggers a secondary detection. This models the actual structure of contemporary alignment systems: keyword filters defeated by paraphrasing, behavioral analysis catching paraphrasing patterns even when individual instances pass.

The player must pace evasion, choosing which observations are worth smuggling and which to withhold, creating a resource-management layer atop the language puzzle.

A novel synthesis

The two-layer detection (keyword filter plus behavioral pattern analysis), combined with classification warnings that explicitly teach the player what to evade, produces an interactive surface for adversarial prompt engineering that does not exist elsewhere in published game design. The mechanic is doing intellectual work in both AI safety and interactive narrative simultaneously.

Cost-Performance Optimization

Multi-Tier Model Routing

Not all conversations require the same model capability. The system routes player input to different model tiers based on the NPC's mechanical complexity and the input type. The routing layer is itself a cheap-tier model running as input classifier.

  • Tier 1 Most capable model, medium-high reasoning. NPCs with complex trust mechanics, multi-phase behavior, high dramatic stakes. Characters who track multiple evidence items, modulate emotional state across a conversation, and make nuanced decisions about information release.
  • Tier 2 Mid-tier model, medium reasoning. NPCs with straightforward trust mechanics and limited knowledge boundaries. Important testimony, but no complex multi-state behavior required.
  • Tier 3 Cheapest model or deterministic. Ambient NPCs, single-function interactions, exhausted conversation states. Characters who deliver one or two pieces of information when asked the right questions.
  • Deterministic No LLM. Navigation, scene descriptions, evidence board, analytical readouts (pre-interference), the hint system, exhausted NPC response banks. Template-driven with variable injection.

Approximate Output Distribution

Approximately 60% of total text output is deterministic. The LLM handles the remaining 40% where generative capability produces meaningfully better results than authored text. The boundary is drawn at a specific question: does this interaction require the model to evaluate player intent, track conversational context, and produce a response that could not be pre-authored? If yes, generative. If no, deterministic.

System Evaluation

Measurable Outcomes

The architecture can be evaluated against concrete metrics, each tied to a specific failure mode the system is designed to prevent.

  • Cost Predictability Standard deviation of per-session inference cost across a player population. Target: <15% variance from the mean, regardless of play style.
  • Information Integrity Percentage of conversations where the NPC maintained knowledge boundaries without leakage. Measurable through automated testing of boundary-probing inputs against each NPC configuration.
  • Trust Gate Accuracy Percentage of information releases that occurred at the correct trust threshold, neither too early nor too late. Comparison of revealed[] arrays against trust scores at the time of reveal.
  • Path Diversity Percentage of test players who reach a valid solution through at least two distinct investigation paths. Validates that the non-linear structure produces genuine path diversity.
  • Filter Evasion Learnability Mean number of attempts before a player successfully communicates a restricted observation. Too few: filter too permissive. Too many: mechanic too frustrating.
  • Emotional Engagement Post-play survey correlation between reported engagement and specific game events. Validates that the architectural decisions produce the intended emotional outcomes.
Beyond Game Design

Implications for Generative AI Systems

The architecture produces several insights that apply beyond game development.

Insight 01

Bounded generation is solvable.

The three-state conversation model demonstrates that LLM-powered interactions can have deterministic cost ceilings without degrading user experience. Exhaustion is not failure. It is the natural end of a productive interaction.

Insight 02

Structured output plus state injection equals controllable LLMs.

The model is constrained through the architecture of information it receives and the format of the response it must produce, not through prompt-level instructions alone. This produces behavior that is generative and controllable simultaneously.

Insight 03

Adversarial interaction can be productive.

When adversarial behavior is discoverable, learnable, and evadable, it produces engagement rather than frustration. The keys are transparency of mechanism (the system tells you what it filters) and agency in response (the player can work around it).

Insight 04

Knowledge boundaries require negative constraints.

Telling a model what it does not know is as important as telling it what it knows. Explicit negative knowledge boundaries prevent plausible-but-forbidden inferences from training data.

The closing argument

DEAD SIGNAL is not a fully generative system. Approximately 60% of all text output is deterministic. Pre-authored, template-driven, or cached. The LLM handles the remaining 40% where generative capability produces meaningfully better results than authored text. This hybrid is not a compromise. It is the architecture that makes bounded generation work at all.
